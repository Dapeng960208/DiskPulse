# 审查问题修复设计

## 目标与范围

修复 2026-07-18 `main` 相对 `origin/main` 审查确认的 7 个可合入问题，按严重度从高到低推进：MySQL 迁移兼容性、项目成员概览范围、Celery 告警投递幂等性、遥测错误码持久化、AI 未知工具恢复、图表异步生命周期和项目成员操作错误提示。

不在本次修复中改动未进入审查问题清单的功能，也不覆盖工作区已有但尚未提交的用户修改。

## 方案选择

1. 推荐方案：每个独立问题先增加能复现缺陷的聚焦测试（RED），再以最小生产代码改动修复（GREEN）。每个 RED 和 GREEN 检查点独立提交；同一根因的跨服务表现合并为一个修复。
2. 备选方案：先集中补测试、再集中修复。该方案减少提交数，但不能逐问题保留 TDD 证据，未采用。
3. 备选方案：仅静态修改并运行全量回归。该方案无法证明测试曾捕获缺陷，未采用。

## 实现边界

- MySQL 迁移：仅在迁移中反射并删除 `projects.pt_user_id` 的外键后删除列；保留 SQLite 和 PostgreSQL 既有路径。
- 首页范围：成员未选择项目时不调用受限全局 Dashboard API，显示明确项目选择状态；超级管理员保留全局视图。
- Celery 告警：用数据库状态与 `next_attempt_at` 作为领取租约，任务领取必须原子化；发送 I/O 保持在事务之外。
- 遥测：仅允许 `failed` 终态保存已枚举的错误码，`success` 和 `skipped` 保持空错误码。
- AI：未知工具调用按失败结果完成审计与 SSE 事件，不访问不存在的 registry 项。
- 图表：异步 ECharts 加载返回后再次确认实例代际和 DOM 状态，过期初始化无副作用。
- 成员管理：将取消确认与 API 失败分开处理，操作失败保留界面状态并显示错误。

所有生产代码的改动处使用简短注释标记所修复的审查问题，避免引入与问题无关的重构。

## 验收与风险

- 每项修复均有先失败、后通过的聚焦测试和相应 Git 检查点。
- 合并前运行相关后端 pytest、前端 Vitest、迁移 SQL 编译检查与构建；完整覆盖率门禁按仓库现有配置执行。
- MySQL DDL 将以离线 SQL 编译和单测验证；本机没有真实 MySQL 服务时，真实升级仍需部署环境复核。
