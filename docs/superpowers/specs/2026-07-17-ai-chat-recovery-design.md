# AI 对话可恢复机制设计

## 目标

避免工具调用上限和模型工具参数格式错误被统一显示为“AI 服务暂不可用”。可恢复场景必须保留工具轨迹与部分结果，给出降级回答，并由用户明确授权后再继续查询。

## 设计

- 工具调用达到既有 `ai.max_tool_iterations` 后，当前回合额外执行一次禁用工具的模型总结。总结必须仅基于当前会话与已返回的工具结果回答，并说明仍需查询的信息；此回合不再执行工具。
- 若总结调用失败，服务端返回不包含敏感工具参数的本地降级说明，而不是通用服务不可用错误。
- 流式工具参数无法解析为 JSON 对象时，客户端保留工具名和安全的错误类别。编排层最多两次向模型反馈格式要求并重试；原始参数不发送给浏览器、审计日志或用户。
- 连续修复失败时同样进入降级完成态，而不是失败态。模型工具参数已通过 JSON 解码但未通过工具输入模型校验时，继续沿用现有受限工具错误反馈路径。
- 降级完成的助手消息使用 `status: "degraded"`，并可选带有 `recovery: { reason, action, label }`。恢复元数据存入既有审计响应载荷，以便历史会话重载后继续展示操作。
- 前端在降级消息下显示恢复原因和“继续查询”或“重新查询”按钮。按钮发送可见的授权消息并创建新的正常受限回合；不会自动突破工具调用上限。
- SSE 客户端必须确认收到 `accepted`，并收到一个终态事件（`completed`、`cancelled` 或 `error`）。传输截断或已知事件载荷异常时保留部分内容、停止流式状态并显示手动重试入口。

## 安全边界

- 不新增可写工具、不放宽路由授权、不绕过按用户限流。
- 工具参数继续由既有动态 Pydantic 输入模型验证。
- 原始非法参数、密钥、令牌和内部异常不进入 SSE、审计展示载荷或用户文案。
- 不新增数据库字段或迁移；使用既有 `AIAuditLog.status` 与 `response_payload`。

## 验收

- 工具上限和修复耗尽均产生可持久化的降级消息与恢复操作，而非“AI 服务暂不可用”。
- 非 JSON 和非对象参数各可自动修复两次；修复成功后继续工具执行。
- 页面刷新后仍可识别降级状态并发起继续查询。
- 正常完成、取消、已有工具校验失败和真实服务故障不回归。
