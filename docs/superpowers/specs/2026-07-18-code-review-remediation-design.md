# 代码审查问题有序修复设计

## 目标

按 `8a66f14a513568c27399ae1153299acfe991807a..main` 审查结果从高到低修复八个问题，并新增 `pnpm mock` 启动入口。每个问题使用独立提交，提交中同时包含复现测试、最小实现和对应跟踪文档。

## 边界与方案

- 路由契约：把事件中心纳入既有懒加载和页面矩阵，不改变路由行为。
- Incident 生命周期：同一关联桶内的新证据可以重开已解决事件；`last_evidence_at` 只允许单调前进。
- 性能资产映射：厂商 UUID/名称只匹配字符串列，只有纯数字标识才参与整数主键匹配。
- 遥测迁移：降级到旧约束前将失败行的分类错误码清空，使已有数据可被旧约束接受。
- AI 配额确认：从持久化审计详情恢复待确认卡片；只向对话所有者返回尚未完成的确认信息，不改变 Redis 一次性消费和超级管理员校验。
- Mock RBAC：系统级资源保持超级管理员专属；项目资源继续使用项目范围能力。
- 中间件日志：发送安全的 500 响应前记录服务端异常与关联 ID，不向客户端泄漏异常。
- Mock 启动：新增 Vite `mock` mode 和 `pnpm mock` 脚本，默认设置 `VITE_USE_MOCKS=true`。

## 数据流与错误处理

所有修复沿用现有 service/router/mock gateway 分层。数据库写入与权限拒绝维持现有状态码；迁移降级只删除新版本无法表达的分类码，不改变运行结果字段；Mock 未授权访问返回 `403`。AI 待确认信息只从审计详情中提取允许展示的 `confirmation_id`、有效期和 preview。

## 测试策略

每项先补聚焦测试并确认 RED，再做最小实现并确认 GREEN。提交前执行 `git diff --check` 和主题相关测试；全部提交完成后执行后端相关测试、前端全量测试、lint、生产构建、Alembic heads 和 Python compileall。
