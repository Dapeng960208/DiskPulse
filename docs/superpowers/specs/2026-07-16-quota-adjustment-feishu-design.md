# 配额调整飞书通知设计

## 目标

项目组或用户目录配额调整成功后，通过现有飞书通知能力告知对应用户，同时不让通知故障回滚已经成功的设备和数据库变更。

## 方案

- 复用配额调整生成的 `StorageAlerts` 记录和现有 `deliver_storage_alert_task`，不新增通知服务或任务。
- 用户目录通知目录用户；项目组通知负责人。收件人继续经过现有 `debug`、全局抄送和超级管理员规则处理。
- 调整记录保存“存储配额调整”标题、对象、旧/新硬限额和旧/新软限额正文，并进入现有 `pending`、`retrying`、`sent`、`failed`、`skipped` 状态流。
- 数据库提交后再入队；入队异常只记录日志，保留待投递记录，由现有重试任务后续处理。
- 现有邮件通知保持不变。

## 验证

- 聚焦测试确认项目组负责人收到完整飞书载荷，调整记录在提交后入队。
- 聚焦测试确认入队异常不影响配额接口成功结果。
- 运行配额调整测试、Python 编译检查和差异检查。
