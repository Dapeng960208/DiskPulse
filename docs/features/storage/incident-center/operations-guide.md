# 事件中心操作指南

## 适用范围与边界

本指南说明“系统管理”的“事件与审计”分区中“事件中心”的派生 Incident。它将容量预测、性能异常、监控质量、DiskPulse 存储告警和厂商系统事件关联为待处置对象；不会删除、替代或混排原始告警和厂商系统事件。

- 原始 DiskPulse 告警仍在“告警”页面。
- NetApp/PowerScale 厂商系统事件仍在存储集群健康分析的“系统事件”页面。
- 事件中心保存不可变证据引用、处置状态、评论和审计轨迹。

## 事件类别与触发条件

| 类别 | 触发条件 | 首先检查什么 |
| --- | --- | --- |
| 容量压力 | 原始存储告警达到 `critical`，或 P90 容量耗尽日期处于未来 7 天内。预测使用至少 30 个有效 UTC 日且覆盖率不低于 80% 的样本。 | 证据摘要中的预测和告警引用、资源限额、已用容量、增长趋势与容量计划。 |
| 设备健康风险 | 已启用且已审核的 `fault_log` 在实例为 `critical` 时进入。目录缺失、待审核或停用的 `critical` 事件可保守进入，但仍标为“未分类厂商事件”；已审核的非故障类型不得进入。非 `critical` 事件仍保留在“系统事件”。 | 先核对事件代码、关联类型和审核状态，再查看可点击的厂商日志正文、事件对象、发生时间和同集群关联资源。 |
| 性能争用 | 仅 P95 延迟可建单：连续 3 个相邻 5 分钟点均为正向退化、每点鲁棒 Z 不低于 `3.5`、每点至少有 12 个过去 28 天同星期同小时样本，且延迟增量均达到 `max(5 ms, 基线 × 50%)`；低延迟、反向变化、样本不足、孤立尖峰或采样空档不触发。IOPS/吞吐量偏离仍保存为诊断观察，不单独建单。MAD 为零时，只有数值不同于基线时使用有界高分。 | P95 延迟、IOPS、吞吐量曲线，证据时间窗、相邻资源和工作负载重叠信息。 |
| 监控盲区 | 本版本不由监控质量快照或采集失败创建 Incident；它们保留在采集账本与健康分析，供判断当前监控可信度。 | 采集运行记录、最后成功时间、覆盖率和数据缺口。该类别是监控盲区，不等同于存储设备发生故障。 |

同一“存储集群 + 资产 + 类别”按距最近证据的滚动窗口归并，默认 `4` 小时（`incident_analytics.correlation_window_hours`）。窗口内跨 30 分钟桶的同类证据仍进入同一事件；超过窗口才创建新事件。已解决事件只在该窗口内获得同类新证据时自动重开；已有历史事件不会自动合并、删除或关闭。

## 容量三级阈值：以哪个为准

默认规则按**硬限额使用率**判定：`important=80%`、`serious=90%`、`emergency=95%`，对应的重复通知间隔为 24、6、1 小时。规则也可以改为软限额口径；资源没有软限额时，该软限额规则不会触发。

对用户目录和项目组等项目范围资源，不应直接拿系统默认 `80/90/95` 作为最终依据。服务端按以下顺序选择一条有效规则：

1. 项目组规则；
2. 项目规则；
3. 系统规则（没有前两者时的兜底）。

因此，当前资源的“有效规则”才是原始存储告警与容量压力事件的告警阈值依据；三道阈值是同一规则中的严重度层级，不是三个需要人工二选一的来源。容量预测是另一条独立信号：它按硬限额的未来耗尽日期生成证据，不会因为使用率尚未达到 `80%` 而停止预警。

原始告警需要连续两次采集均越过当前等级才首次触发；之后越级会立即升级，并按该等级的重复间隔提醒。`important`（80%）和 `serious`（90%）仍按原告警规则通知，但不创建 Incident；只有 `emergency`（95%，即原始记录 `critical`）可作为容量压力的直接事件依据。事件详情中的“已确认/调查中”等人工状态当前**没有**超时 SLA 或“未处理自动升档”机制。派生 Incident 通知仅在新建、系统重开或严重度升级时发送，原始告警的重复提醒不会反复创建新 Incident。

## 证据、分数与置信度

事件列表、关联证据和时间线都按事件发生时间由近到远展示；事件列表以最近证据时间作为排序依据，编辑严重度、状态或认领信息不会把旧事件移到最新事件之前。详情先以“关联概览”合并同类依据，再在“关联证据”中显示每条证据的异常说明、影响范围和发现时间。例如，三条容量、性能和厂商事件采集的监控质量异常会汇总为“监控可用性异常”，展开后分别显示“容量采集已过期”“性能采集覆盖不足”等结论。默认视图不展示拼接的内部标识。

性能异常证据会直接说明具体指标（P95 总延迟、P95 IOPS 或 P95 吞吐量）、连续三个相邻 5 分钟窗口覆盖的时间范围、窗口末点 P95、过去 28 天同星期同小时样本的季节基线、正常参考范围和鲁棒 Z 分数。当前异常观测只保存连续窗口的末点数值，不伪造其余两个窗口的数值。正常参考范围按基线两侧 `3.5 × MAD / 0.67448975` 计算，下界按非负性能指标收敛到零；MAD 为零时上下界都等于基线。

`source_ref` 仍是用于回查原始事实的不可变引用；例如 `forecast:*` 来自容量预测，`alert:*` 来自 DiskPulse 存储告警，`anomaly:<id>` 的数字部分是 `anomaly_observations.id`。它只在证据项可展开的“技术关联信息”中显示，供排障复制；该区域只展示“证据标识、标识作用、回查方式”，不再重复上方已有的关联对象、证据范围和证据类型。性能异常可在 `GET /storage-pulse/api/v1/anomalies` 中按 `metric`、`observed_from`、`observed_to` 和存储集群缩小范围，再以返回的 `id` 精确核对。其他标识按前缀回查对应预测、告警、厂商事件或监控记录；没有稳定路由或权限保证的原始记录不会生成伪造跳转。容量压力的告警证据会标明 `hard_limit_alert` 或 `soft_limit_alert`，分别对应有效规则使用的硬/软限额口径。

时间线显示本地可读时间、事件、说明和操作人。系统自动写入的 `created` 显示为“系统创建事件”，`evidence_added` 显示为“关联新证据”；新证据会附带简短说明。历史时间线若没有已存储说明，则显示中性系统说明而不直出英文内部事件名。

“确定性诊断”由服务端固定规则计算，不是 AI 的自由文本判断。系统按类别挑选最高权重证据、扣除冲突证据，并按独立证据类型数、最新证据时间和固定优先级排序。高置信度还要求部署配置 `incident_analytics.replay_gate_verified=true`、候选分数至少 `0.8`、至少两类独立证据；任一条件不满足时会显示为 `medium`、`low` 或 `insufficient`。

`asset_mapping_missing` 在页面显示为“资产映射不完整”。它表示事件至少已归属存储集群，但节点、卷、Qtree 或项目的稳定映射链路不完整；厂商事件已有稳定节点身份时不会产生该缺口。它不是“没有采集到日志”，也不是“无法识别事件代码”。排查时仍可从存储集群“系统事件”点击“查看日志”，核对事件代码、对象、规范化日志和发生时间。

## 状态与处置

状态只能相邻推进：`open → acknowledged → investigating → mitigated → resolved`。

| 状态 | 含义 | 推荐动作 |
| --- | --- | --- |
| `open` | 系统已创建，尚未确认。 | 查看证据并认领。 |
| `acknowledged` | 已确认收到。 | 记录初步判断，开始核查。 |
| `investigating` | 正在调查。 | 对照原始告警、厂商事件、预测和监控数据。 |
| `mitigated` | 影响已缓解。 | 持续观察是否有新证据。 |
| `resolved` | 处理完成。 | 写入处理结论和后续复盘项。 |

- **认领**：将事件指派给当前用户，明确处理责任。
- **释放**：取消当前认领；仅认领人或超级管理员可执行。
- **取消静默**：恢复派生事件的后续通知；不会删除事件、证据、原始告警或厂商事件。
- **创建维护窗口**：按项目、可选集群/资产和 UTC 时间范围抑制相关派生事件的创建、重开与通知；采集、分析快照和原始事实继续保留。
- **评论**：记录判断、操作、交接和结论。上述写操作均写入统一审计。

`reader` 只能查看项目范围内的事件、诊断和证据摘要；`editor` 可以认领、推进、静默和评论；`project_admin` 可创建本项目维护窗口；无项目范围事件仅 `super_admin` 可操作。

## 厂商事件关联与历史修复

“系统管理”的“事件与审计”分区中，“厂商事件关联目录”维护 NetApp/PowerScale 事件代码的中文含义和关联类型。关联类型包括故障日志、性能异常、容量/配额阈值、系统运行事件、监控能力下降和未分类厂商事件。只有已启用且已审核定义影响业务；已审核定义必须使用非 `unknown` 类型，并有 HTTPS 官方链接和版本范围。待审核项只在管理目录中展示，不形成正式语义或重复故障。目录只解释事件，不删除或屏蔽采集结果；未知代码会作为待审核项保存，管理员必须依据官网或设备运行时目录审核，不能按名称猜测。

存储集群“系统事件”显示所有厂商事件及其安全关联信息；“重复故障”只显示已启用且已审核、关联类型为“故障日志”且指纹重复的记录。故障指纹只用于归组，默认不作为故障结论；点击“查看日志”可查看具体日志正文、事件对象和发生时间。

升级后由超级管理员人工执行一次“发现已有代码”：系统幂等写入基础定义，仅扫描历史 `storage_alerts` 为未收录代码创建 `unknown + pending` 项，并返回新建、已存在和历史事件修复计数。它不是周期任务，也不主动查询存储设备事件目录。旧版本误建的非 `critical` 厂商 `device_fault` Incident 会被标记为已解决并追加兼容修复时间线；原始事件、证据和诊断不会删除。重复执行不会重复创建占位项、关闭 Incident 或写时间线。详细规则见[厂商事件关联目录](../event-association/overview.md)。

注意一处升级期边界：本版本起，带稳定节点身份的厂商事件按节点（`storage_node`）而非集群聚合。升级前仍处于打开状态的 `critical` 厂商 `device_fault` Incident（集群级关联键）不会被自动复核关闭（复核仅处理全部证据为非 `critical` 的情形），其后续同类事件会开出新的节点级 Incident。运维应在升级后检查打开中的集群级厂商故障 Incident，确认故障已被节点级事件接管后手工解决旧事件。
