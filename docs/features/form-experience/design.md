# 写入表单体验重构设计

> 状态：仅设计，待实现
> 确认日期：2026-07-16

## 1. 背景

DiskPulse 当前写入表单主要基于 Element Plus，并在多个资源弹窗中复用 `useForm` 和 `useDialog`。现有实现已具备基本提交能力，但不同页面在标签位置、弹窗宽度、字段间距、按钮文案、失败反馈和复杂字段分组上不一致；`useForm` 还缺少脏状态、未保存关闭提醒和校验失败后的首错定位。

本设计统一所有当前写入入口的视觉和交互，同时保持现有字段语义、权限、API、payload 与业务流程不变。

## 2. 目标与非目标

### 2.1 目标

- 建立统一、具有品牌识别度的写入表单视觉规范。
- 简单表单保持紧凑，复杂表单按语义分组并提高填写效率。
- 统一校验、提交、成功、失败、未保存关闭和高风险确认行为。
- 保持键盘操作、焦点反馈和窄屏可用性。
- 复用 Element Plus、`useForm`、`useDialog` 和现有设计变量，不新增依赖。

### 2.2 非目标

- 不修改后端接口、权限、数据库或字段含义。
- 不删除、合并或新增业务字段。
- 不引入 Schema 驱动动态表单、表单框架或新组件库。
- 不重构登录、查询筛选和导出表单。
- 不把当前不存在的扩容写入表单作为本轮新增功能；未来新增时遵循本规范。

## 3. 当前覆盖范围

### 3.1 创建与编辑

- 存储集群：`StorageClusterFormDialog.vue`
- 用户：`UserFormDialog.vue`
- 项目：`ProjectFormDialog.vue`
- 项目组：`GroupFormDialog.vue`
- 项目组标签：`GroupTagFormDialog.vue`
- 用户目录：`UsageFormDialog.vue`
- AI 模型配置：`AiCenterPage.vue` 内模型弹窗
- 系统设置：`SettingsPage.vue` 页面内保存表单

### 3.2 高风险写入确认

- 存储集群、用户、项目组和项目组标签删除。
- 用户目录移动到备份目录。
- 备份删除与回滚。
- LDAP 用户同步。

## 4. 方案选择

### 4.1 已选方案

采用“共享样式 + 增强现有 `useForm` / `useDialog`”。每个业务表单继续保留明确的 Vue 模板和 API 调用，共享层只负责一致的生命周期、状态与视觉规则。

该方案比仅做全局 CSS 更能覆盖交互问题，也避免 Schema 动态表单带来的描述层、动态插槽和调试成本。

### 4.2 不采用的方案

- 仅全局 CSS：无法可靠统一字段分组、首错定位和未保存提醒。
- Schema 驱动动态表单：当前表单数量和差异不足以抵消额外抽象成本。
- 全部改为抽屉：会同时维护弹窗与抽屉两套交互规则，且当前页面上下文没有必须持续可见的需求。

## 5. 结构设计

### 5.1 `useForm`

在保持现有调用方式兼容的前提下补充：

- 打开或重置表单时保存初始模型快照。
- 暴露只读脏状态，用于判断内容是否实际变化。
- 提交前执行完整校验；失败后滚动并聚焦第一个无效字段。
- 提交期间保持单一 `submitting` 状态，阻止重复提交。
- 成功后重置快照，避免正常关闭触发未保存提醒。
- 保持 API 失败向上传播或进入既有回调，不在共享层重复显示通用错误。

快照比较只服务当前小型表单模型，优先使用可预测的序列化比较，不建设通用深度差异引擎。

### 5.2 `useDialog`

保留现有 `open`、`close` 和 `visible`，增加可选的关闭守卫：

- 未修改时直接关闭。
- 已修改时，通过取消按钮、关闭按钮、遮罩或 `Esc` 关闭都进入同一确认流程。
- 提交期间拒绝关闭。
- 成功提交后的程序化关闭绕过未保存确认。

### 5.3 共享样式

先在现有设计变量中补齐必要 token，再由全局表单样式引用。业务组件不新增散落颜色值。

共享样式只定义：

- 品牌渐变标题区、标题和简短说明。
- 顶部字段标签、必填标记、帮助文字和错误文字。
- 单列与双列网格、语义分组、底部操作区。
- 输入、禁用、聚焦、错误和提交中状态。
- 桌面、平板和手机的响应式回落规则。

业务表单显式声明字段分组，不新增字段配置 DSL。

## 6. 视觉规范

采用已确认的“品牌强化”方向：

- 弹窗标题区使用蓝色品牌渐变，标题下方只放当前操作所需的简短说明。
- 输入区域使用浅蓝背景和克制边框；聚焦时使用清晰的蓝色外环。
- 分组标题使用品牌蓝建立层级，不叠加无意义卡片。
- 错误状态同时使用边框、文字和焦点，不只依赖颜色。
- 底部操作区使用浅蓝背景，与正文形成稳定边界。
- 主按钮文案描述实际动作，例如“创建集群”“保存修改”，不统一写成含义模糊的“提交”。

布局规则：

- 1–4 个字段的简单表单使用紧凑单列。
- 复杂表单使用宽弹窗、语义分组和双列网格。
- 文本域、告警规则、账号帮助等需要完整宽度的内容跨两列。
- 容器不足时自动回落单列，页面和弹窗不得产生横向滚动。
- 系统设置使用页面内分组与固定保存操作区，不套弹窗。

## 7. 交互与数据流

### 7.1 打开

1. 创建模式加载默认模型；编辑模式复制当前记录。
2. 建立初始快照并清除旧校验状态。
3. 弹窗打开后聚焦第一个可编辑字段。

### 7.2 填写与校验

- 标签位于控件上方，必填字段显示 `*`。
- 关键规则使用字段下方帮助文字，不只写在 placeholder 中。
- 字段失焦后显示就地错误；值恢复有效后及时清除。
- 关联字段变化时，只清空确定失效的下级值，并说明清空原因。
- 提交校验失败时滚动并聚焦第一个错误字段。

### 7.3 提交

1. 用户点击具体动作按钮。
2. 完整校验通过后进入提交中状态。
3. 按钮显示“创建中…”或“保存中…”，并阻止重复提交和关闭。
4. 成功后显示具体结果、更新快照、关闭表单并刷新来源列表。
5. 失败后保留用户输入和弹窗状态，便于修正或重试。

### 7.4 错误反馈

- 网络、权限和通用服务端错误继续由 `request-builder.js` 统一显示。
- 只有服务端返回稳定字段映射时，才把错误落到对应字段。
- 同一失败不得同时出现共享 toast 和表单本地重复 toast。
- 失败提示不得泄露凭据、token、内部路径或栈信息。

### 7.5 关闭与高风险确认

- 未修改表单直接关闭。
- 已修改表单关闭时提示“放弃未保存的修改？”，默认动作是“继续编辑”。
- 普通创建和修改提交不增加二次确认。
- 删除、备份移动、备份回滚、LDAP 同步等高风险操作必须说明对象、影响和是否可撤销；确认按钮使用具体动词。

## 8. 可访问性

- 所有控件保留键盘访问和 Element Plus 原生语义。
- 聚焦状态必须具有可见的 `focus-visible` 表达。
- 帮助与错误文字通过 `aria-describedby` 关联到字段。
- 弹窗关闭后将焦点返回原触发按钮。
- 错误摘要或错误计数不得替代字段旁的具体错误。

## 9. 测试与迁移策略

实施必须使用 TDD：

1. 先为 `useForm` 和 `useDialog` 增加失败测试，覆盖脏状态、关闭守卫、首错定位、重复提交和成功重置。
2. 实现共享行为和样式。
3. 按“简单 CRUD、复杂 CRUD、AI 模型、系统设置、高风险确认”迁移，并逐组运行聚焦测试。
4. 最后执行前端全量单测、lint、生产构建以及桌面和窄屏人工检查。

Portal 组件测试必须挂载到 `document.body` 并在用例后清理。共享 composable 修改需要验证全部现有调用者，不只验证一个示例表单。

## 10. 验收标准

- 所有范围内写入表单采用统一品牌主题、顶部标签和具体操作文案。
- 简单表单为紧凑单列；复杂表单为分组双列；窄屏自动单列且无横向滚动。
- 校验错误可见、可定位、可聚焦且不只依赖颜色。
- 未修改关闭不提醒，实际修改后关闭才提醒。
- 提交期间不能重复提交或误关闭。
- 成功和失败提示不重复，失败后保留输入。
- 高风险确认明确对象、影响和可撤销性。
- 字段语义、payload、API、权限和业务结果保持不变。
- 不新增 npm 依赖，不引入 Schema 表单或新组件库。

## 11. 风险与控制

- `useForm` 是多个弹窗的共享依赖：保持现有返回字段兼容，并用全部调用者测试控制回归。
- 全局样式可能影响查询或登录表单：共享规则必须挂在写入表单专用根类下，不做无范围的 Element Plus 覆盖。
- 脏状态可能受对象顺序或派生字段影响：快照只包含表单可写模型，派生展示值不参与比较。
- 服务端错误结构不完全一致：没有稳定字段映射时维持全局错误提示，不猜测字段归属。
