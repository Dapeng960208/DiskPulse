# 开发错误复盘规范（AI 快速版）

本规范用于判断何时把错误写入 `docs/tracking/error-log.md`。

## 1. 必须记录

- 构建、测试、迁移、脚本、CI 命令失败。
- Python/Node 版本、依赖、权限、路径、编码导致的环境失败。
- 配置缺失或配置错误。
- API、路由、权限不一致导致的 404/401/403/501 等。
- 文档与实现不一致导致误用或返工。

明显拼写错误且 1 分钟内修复、没有复现价值的误输入，可以不记录。

## 2. 固定入口

- 错误库：`docs/tracking/error-log.md`。
- 不要只写在聊天记录或提交信息中。

## 3. 最小模板

```md
### YYYY-MM-DD：一句话概括
- 触发：执行了什么操作或命令。
- 现象：关键报错或症状，摘关键行即可。
- 根因：确认后的原因。
- 修复：改了什么文件、配置或命令。
- 验证：如何确认已修复。
- 风险：仍未验证或可能复发的范围。
```

## 4. 联动更新

- 错误暴露规范缺口时，更新对应 `docs/standards/`。
- 错误涉及功能、接口、配置、权限、迁移或部署时，更新对应正式文档。

## 5. 列表分页类问题的防错规则

- 核心原因通常不是“少写了一个分页参数”，而是把接口职责混淆了：后端列表接口已经按数据库分页返回时，前端还沿用“本地持有全量数据再切页”的思路，就会出现只拿到默认第一页、`total` 不准、翻页无效或页面显示一页的错觉。
- 遇到后端分页接口，前端分页组件必须直接驱动 `page` / `page_size`，`total` 只能来自同一过滤条件下的后端 `count` 结果，不能靠已加载行数、默认首屏条数或前端临时补算。
- 列表筛选如果会影响后端查询成本，优先采用“草稿编辑 + 显式搜索”模式，避免输入变化立即触发请求，也避免筛选状态和分页状态彼此打架。
- 如果页面同时有页签头、筛选栏、表格和分页栏，先确认共享布局壳是否已经承担了间距与底部节奏，不要再插入重复的标题数量行或额外容器，防止分页栏看起来“贴边”“错位”或超出表格节奏。
- 复盘这类问题时，先写清楚“分页职责在后端还是前端”“总数从哪里来”“筛选何时生效”三个问题，再动手改实现和测试。
